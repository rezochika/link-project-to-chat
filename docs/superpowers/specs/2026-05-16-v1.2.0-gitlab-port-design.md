# v1.2.0 — GitLab parity sweep (safety prompt + hot-reload + group context + meta_dir)

**Date:** 2026-05-16
**Status:** Approved (brainstorming complete; awaiting plan)
**Author:** Revaz Chikashua (drafted with Claude)
**Target release:** v1.2.0 (additive on top of `dev` post-v1.1.1)

## Summary

Four small ports from the GitLab fork (`~/PycharmProjects/gl/link-project-to-chat`, v0.16.0) that close real operational gaps in the GitHub version. None of the four touches the transport-abstraction or backend-abstraction boundaries — they extend them.

1. **Safety system prompt.** A hardcoded GitLab guardrail asking Claude to describe-and-ask before destructive action, ported as a per-project opt-out (`ProjectConfig.safety_prompt`) and threaded through `BaseBackend.safety_system_prompt` so every current and future backend renders it natively.
2. **Hot-reload of `allowed_users`.** Project bot and manager bot pick up `allowed_users` changes from disk within 5 s of the manager writing them, no restart required. Lives in `AuthMixin` so the same code serves both bots; cross-transport because auth is.
3. **Inter-message group context.** When the bot makes an LLM call in a group, prepend the chatter that happened since its last LLM call. New `ChatHistory` module keyed on transport-portable `ChatRef`; per-call kwarg through `submit_agent → Task → chat_stream`; per-backend rendering in the backend's native style (Claude's `--append-system-prompt`, Codex's `<system-reminder>` block, etc.). Always on for groups; DMs continue to use `conversation_log`.
4. **`meta_dir` config field.** Let operators put per-bot / per-plugin persistent storage on a different volume than `config.json`. Plugin-side machinery is already in `plugin.py`; this just adds the config seam.

Out of scope (already done in GitHub): plugin `commands()` + `callbacks()` (BotCommand + `on_button` consume-or-passthrough already wired).

## Background

The GitLab fork is the merge source for the v1.0.0 plugin port and the v1.1.0 `respond_in_groups` port. Both v1.0.0 and v1.1.0 took specific subsets; this v1.2.0 sweeps up the remaining operational-quality items. Three of the four (#1, #2, #3) address concrete pain points surfaced during recent debugging sessions:

- **#1** — The 2026-05-16 debug session (chat export `ChatExport_2026-05-16/result.json`) shows the bot executing `kill -9` on the manager process from an ambiguous "can you restart the service?" request. The GitLab guardrail would have triaged this into "I'd describe the restart options; do you want me to run any of them?"
- **#2** — Same session shows the user repeatedly running `scripts/restart.sh` after every `/add_user` call in the manager. With hot-reload, those restarts disappear.
- **#3** — The user's group `R.Chikashua, Rezo and link-project-to-chat Claude` has human chatter between LLM invocations. Without inter-message context, the bot loses that history; with it, the bot sees `[Recent discussion]` and can answer "what did Rezo just ask?"
- **#4** — Operator-requested separation of config and data volumes. Already on the GitLab fork; small lift to port.

Reference: GitLab v0.16.0 commits `3ec8b9b` (#1), `ae3c1dd` (#2, #3), `5987654` (#4).

## Non-goals

- **Replacing `conversation_log`.** v1.2.0 keeps `conversation_log` for DM turn pairs. `ChatHistory` is groups-only.
- **Persisting chat history across restarts.** In-memory matches GitLab. Future persistence is YAGNI for now; can be layered without breaking the API.
- **Plugin `callbacks() -> dict[str, Callable]`.** GitHub's `Plugin.on_button(click) -> bool` consume-or-passthrough is a different design choice, not a missing feature.
- **Migrating existing plugin data.** No automatic move from `~/.link-project-to-chat/meta/` to a custom `meta_dir`. Operators who set the field also move the directory.

## Design

### Sub-feature 1 — Safety system prompt

**Goal:** Every Claude / Codex / future-backend invocation includes a guardrail asking the agent to describe-and-ask before destructive action, unless an operator explicitly disabled it for that project.

**Data model:**
- `BaseBackend.safety_system_prompt: str | None` — instance field, default `None` (= use module default).
- Module-level constant `DEFAULT_SAFETY_SYSTEM_PROMPT` in `backends/base.py`, holding the GitLab guardrail text verbatim:

  > `<important>Only make changes or run commands when explicitly asked to modify a specific file or perform a specific task. For questions, analysis, or discussion — answer only, do not act. If you identify something that could be fixed or improved, describe what and why, then ask for approval before doing anything. Do not run, start, stop, or restart anything unless explicitly asked. Do not install packages, run scripts, or restart services unless explicitly asked in the current message.</important>`

- `ProjectConfig.safety_prompt: str | None = None` — config field, three semantics:
  - Key absent OR `null` → use `DEFAULT_SAFETY_SYSTEM_PROMPT` (safety ON, default text)
  - Non-empty string → use that exact string (safety ON, custom text)
  - Empty string `""` → disable safety (safety OFF)

**Bot integration** (`bot.py:_build_backend`):
```python
# Resolve safety prompt: empty string means disabled; None means default.
if cfg.safety_prompt == "":
    backend.safety_system_prompt = ""  # explicitly disabled
elif cfg.safety_prompt is None:
    backend.safety_system_prompt = DEFAULT_SAFETY_SYSTEM_PROMPT
else:
    backend.safety_system_prompt = cfg.safety_prompt
```

**Per-backend rendering:**

Claude (`backends/claude.py:chat_stream` → `_build_cmd(user_message, recent_discussion)`):
```python
# _build_cmd accepts recent_discussion as a parameter (not from self)
parts = [_build_telegram_awareness(self.capabilities), _ASK_DISMISSED_HINT]
if self.safety_system_prompt:    # instance state (per-bot)
    parts.append(self.safety_system_prompt)
if recent_discussion:            # per-call kwarg (per-message)
    parts.append(recent_discussion)
if self.team_system_note:        # instance state (per-bot)
    parts.append(self.team_system_note)
if self.append_system_prompt:    # instance state (per-bot)
    parts.append(self.append_system_prompt)
cmd.extend(["--append-system-prompt", "\n\n".join(parts)])
```

Codex (`backends/codex.py:chat_stream` → `_build_prompt(user_message, recent_discussion)`):
```python
blocks = []
if self.safety_system_prompt:
    blocks.append(f"<system-reminder>\n{self.safety_system_prompt}\n</system-reminder>")
if recent_discussion:
    blocks.append(f"<system-reminder>\n{recent_discussion}\n</system-reminder>")
if self.team_system_note:
    blocks.append(f"<system-reminder>\n{self.team_system_note}\n</system-reminder>")
if not blocks:
    return user_message
return "\n\n".join(blocks) + "\n\n" + user_message
```

Order in final prompt is consistent across backends: safety → recent_discussion → team → user message.

`safety_system_prompt`, `team_system_note`, `append_system_prompt` remain **instance fields** on the backend (per-bot, set at startup or team change). `recent_discussion` is a **per-call kwarg** (per-message, no shared state).

**Future backends:** inherit `safety_system_prompt`; each implements rendering once in its own `_build_cmd` / `_build_prompt`. Standard pattern documented next to `team_system_note`.

**Tests:**
- `BaseBackend.safety_system_prompt` default behavior (None → default text, "" → disabled, custom → custom).
- Claude backend: `--append-system-prompt` payload contains the safety text by default; absent when `safety_prompt=""`.
- Codex backend: `<system-reminder>` block present by default; absent when disabled.
- Config loader: round-trip with default, custom string, explicit `None`, empty string.
- CLI: `projects add --safety-prompt "..."` and `projects edit NAME safety_prompt ...`.
- Manager bot: `safety_prompt` field appears in the project-edit keyboard with a multi-line text picker (no toggle here — too much value in the custom-string path to coerce to On/Off).

### Sub-feature 2 — Hot-reload `allowed_users`

**Goal:** When the manager bot writes a new `AllowedUser` to disk (via `/add_user`, `/promote_user`, etc.), the running project bot picks it up within 5 s. No restart required.

**Trigger:** debounced polling on every `_auth_identity` call. 5-second floor between disk reads. Reads are cheap (single config-file `mtime` + parse on cache miss); the floor keeps it cheap under flooding.

**Hot-reload contract** (added to `_auth.py`):
```python
class AuthMixin:
    def _reload_allowed_users_if_stale(self) -> None:
        """Refresh self._allowed_users from disk, debounced to 5 s.

        Pre-conditions:
          - Self has _last_allowed_users_reload, _project_config_path,
            _project_name (project bot) or just the global path
            (manager bot).
          - Self has _auth_dirty.

        Behavior:
          - No-op if _auth_dirty is True (we have unsaved changes; reloading
            would clobber in-flight first-contact locks).
          - No-op if 5 s haven't elapsed since last reload.
          - Else: load_config(path); read project- or global-scope
            allowed_users; merge with current in-memory state by username,
            preserving locked_identities union; replace self._allowed_users.
          - On any exception: log warning, keep current state, set
            _last_allowed_users_reload to now (retry in 5 s).
        """
```

**Call site** (`_auth.py:_auth_identity` and `_require_executor`):
```python
def _auth_identity(self, identity) -> bool:
    self._reload_allowed_users_if_stale()  # new line
    if not self._allowed_users:
        return False
    # ... existing body unchanged ...
```

**Merge semantics** (per username):
- If disk has the username and memory has it: take disk's `role`; `locked_identities` is the **union** of disk + memory. Preserves any in-flight first-contact lock that hasn't been persisted yet.
- If disk has it and memory doesn't: append (new user added by manager).
- If memory has it and disk doesn't: drop (operator removed user via `/remove_user`).
- Same-transport spoof guard still applies on subsequent auth calls; reload doesn't reset it.

**Concurrent-write safety:**
- Manager bot writes through `locked_config_rmw` (fcntl.flock-protected RMW).
- Project bot reads via `load_config` (no lock; reads the post-fsync state).
- Worst case: project bot reads an in-flight RMW's pre-fsync state → reads next time (within 5 s) get the post-fsync state. No data corruption.

**ManagerBot path:** same `_reload_allowed_users_if_stale` API, different field source (`config.allowed_users` for the manager's global scope). Manager doesn't auto-reload its `projects[*].allowed_users` because the manager doesn't auth against those — they belong to the project bots.

**Failure modes:**
- Config file missing → log + keep current state.
- Config file unparseable → log + keep current state.
- Disk role conflicts with memory role on a locked identity → disk wins. The lock binding persists; only role changes.

**Cross-transport / cross-backend guarantees:**
- All auth-keyed state (brute-force counter, rate-limits) is process-local and keyed on `_identity_key(identity) = "transport_id:native_id"`. Hot-reload doesn't reset these (deliberate — a reload doesn't earn a spoofer another 5 attempts).
- AuthMixin is a pure-Python mixin with no transport or backend imports.

**Tests:**
- Bot starts with `[alice executor]`; write `[alice executor, bob viewer]` to disk; within 5 s, next auth for bob succeeds with role=viewer.
- Bot has `_auth_dirty=True` (mid-first-contact); reload skipped; on next clean cycle the persist+reload converge.
- Disk has corrupt JSON; reload no-ops; auth still works against cached users.
- Concurrent manager `locked_config_rmw` + bot `_reload_allowed_users_if_stale`: no torn read.
- Manager bot's own hot-reload path (different scope: global `allowed_users`).
- Locked-identities union preserves in-flight first-contact lock through a reload.

### Sub-feature 3 — Inter-message group context

**Goal:** When the bot makes an LLM call in a ROOM-kind chat, prepend the chatter that happened in that chat since its last LLM call. Groups only; DMs continue to use `conversation_log`.

**New module: `src/link_project_to_chat/chat_history.py`** (≈80 lines):

```python
"""In-memory chat history per ROOM. Process-local; not persisted.

Keyed on transport-portable ChatRef so the same class works for
Telegram groups, future Web rooms, Slack channels, Google Chat rooms.

Zero telegram imports. Zero backend imports. Verified by extending
tests/test_transport_lockout.py to lock down this module.
"""
from __future__ import annotations

import collections
import logging
from typing import Iterable

from .transport.base import ChatKind, ChatRef

logger = logging.getLogger(__name__)

_DEFAULT_MAXLEN = 200                  # Per-chat ring buffer size; matches GitLab.
_DEFAULT_MAX_SERIALIZED_CHARS = 4000   # Defensive cap on injected text.
_TRUNCATION_PREFIX = "[…older messages truncated…]\n"


class ChatHistory:
    def __init__(
        self,
        maxlen: int = _DEFAULT_MAXLEN,
        max_serialized_chars: int = _DEFAULT_MAX_SERIALIZED_CHARS,
    ) -> None:
        self._history: dict[ChatRef, collections.deque[dict]] = {}
        self._last_llm_msg_id: dict[ChatRef, str] = {}
        self._maxlen = maxlen
        self._max_chars = max_serialized_chars

    def record(
        self,
        chat: ChatRef,
        msg_id: str,
        sender: str,
        text: str,
        *,
        is_own_bot: bool = False,
    ) -> None:
        """Append a message to the per-chat buffer.

        Skips when chat.kind != ROOM, when text is empty after strip,
        when is_own_bot is True. Caller passes is_own_bot when sender is
        the bot itself (own outbound messages don't count as chatter).
        """
        if chat.kind != ChatKind.ROOM:
            return
        if is_own_bot:
            return
        if not text or not text.strip():
            return
        dq = self._history.setdefault(chat, collections.deque(maxlen=self._maxlen))
        dq.append({"msg_id": msg_id, "sender": sender or "unknown", "text": text})

    def since_last_llm(self, chat: ChatRef, before_msg_id: str) -> str:
        """Serialize messages between last_llm_msg_id (exclusive) and
        before_msg_id (exclusive). Returns "" if no messages between.

        Truncates oldest if serialized length > max_serialized_chars.
        """
        dq = self._history.get(chat)
        if not dq:
            return ""
        last_llm = self._last_llm_msg_id.get(chat)
        msgs: list[dict] = []
        passed_last = last_llm is None
        for entry in dq:
            if not passed_last:
                if entry["msg_id"] == last_llm:
                    passed_last = True
                continue
            if entry["msg_id"] == before_msg_id:
                break
            msgs.append(entry)
        if not msgs:
            return ""
        return self._serialize(msgs)

    def mark_llm_call(self, chat: ChatRef, msg_id: str) -> None:
        """Record that an LLM call started for msg_id. Subsequent
        since_last_llm calls won't include earlier messages."""
        self._last_llm_msg_id[chat] = msg_id

    def _serialize(self, msgs: Iterable[dict]) -> str:
        lines = [f"{m['sender']}: {m['text']}" for m in msgs]
        body = "\n".join(lines)
        if len(body) <= self._max_chars:
            return f"[Recent discussion]\n{body}\n\n"
        # Drop oldest until fit; prepend truncation marker.
        while lines and len("\n".join(lines)) > self._max_chars:
            lines.pop(0)
        body = "\n".join(lines)
        return f"[Recent discussion]\n{_TRUNCATION_PREFIX}{body}\n\n"
```

**Per-call plumbing (not an instance field):** unlike `safety_system_prompt` and `team_system_note` (which are per-bot and set at startup), `recent_discussion` is **per LLM invocation** — different for every chat-message that triggers a backend call. The shared backend instance can't hold it without races (PTB's `concurrent_updates=True` allows two chats' handlers to interleave between `submit_agent` and the actual `chat_stream` call).

Threading it through the right layers:

```python
# task_manager.py
@dataclass
class Task:
    ...
    recent_discussion: str = ""    # NEW

def submit_agent(
    self,
    chat: ChatRef,
    message: MessageRef,
    prompt: str,
    *,
    recent_discussion: str = "",   # NEW
) -> Task:
    task = Task(..., recent_discussion=recent_discussion)
    return self._submit(task)

async def _run_agent_turn(self, task: Task, ...) -> None:
    ...
    async for event in active_backend.chat_stream(
        task.input,
        recent_discussion=task.recent_discussion,   # NEW
        on_proc=...,
    ):
        ...

# backends/base.py — Protocol update
class AgentBackend(Protocol):
    async def chat_stream(
        self,
        user_message: str,
        *,
        recent_discussion: str = "",   # NEW
        on_proc: Callable | None = None,
    ) -> AsyncIterator[StreamEvent]: ...
```

Each backend reads `recent_discussion` as a local kwarg inside its `chat_stream`, not as instance state. No shared-state race.

**Bot integration** (`bot.py` `__init__`):
```python
self._chat_history: ChatHistory = ChatHistory()
```

**Recording call sites:**
- `_on_text_from_transport` — after the gate decisions, before backend dispatch:
  ```python
  self._chat_history.record(
      chat=incoming.chat,
      msg_id=incoming.message.native_id,
      sender=(incoming.sender.handle or incoming.sender.display_name or "unknown"),
      text=incoming.text,
      is_own_bot=is_from_self(incoming, self.bot_username),
  )
  ```
- `_on_file_from_transport` — same, using `incoming.text` as the caption.
- `_on_voice_from_transport` — same. (Empty-text guard inside `record` handles voice-without-caption case.)

**Injection call site** — in `bot.py`, every place that calls `self.task_manager.submit_agent(...)` (currently 4 call sites: `bot.py:1122`, `:1373`, `:2731`, `:2795`) gets a small helper that resolves `recent_discussion` based on `chat.kind`:

```python
# bot.py helper, defined once
def _resolve_recent_discussion(self, incoming) -> str:
    if incoming.chat.kind != ChatKind.ROOM:
        return ""
    text = self._chat_history.since_last_llm(
        incoming.chat, incoming.message.native_id
    )
    self._chat_history.mark_llm_call(
        incoming.chat, incoming.message.native_id
    )
    return text

# At each submit_agent call site
self.task_manager.submit_agent(
    chat=incoming.chat,
    message=incoming.message,
    prompt=prompt,
    recent_discussion=self._resolve_recent_discussion(incoming),
)
```

DMs always get `""` for `recent_discussion` (DMs use `conversation_log` for prior-turn context, not chat-message history).

**Cross-transport recording:** `ChatHistory.record` uses only transport-portable types (`ChatRef`, `ChatKind`, `str`). The bot recording sites use only `IncomingMessage` fields. Same code paths fire for every transport that surfaces `ChatKind.ROOM`. Current production Web UI dispatches direct browser chats as `ChatKind.DM`, so Web room context is deferred until Web grows a real multi-user room model.

**Cross-backend rendering:** the `recent_discussion` string is passed as a per-call kwarg into `chat_stream`; each backend's `_build_cmd` / `_build_prompt` decides how to surface it (Claude folds it into `parts`, Codex prepends a `<system-reminder>` block). Future Gemini backend implements the kwarg in its own `chat_stream`.

**Privacy posture:**
- All senders are recorded — operator's explicit "match GitLab" choice.
- Process-local in-memory only; never persisted.
- Cleared on bot restart.
- Reaches the LLM via the backend's normal prompt path; same trust boundary as any other backend input.

**Edge cases handled:**
- DM message → not recorded (recording is room-only).
- Sticker / voice with no caption → not recorded (empty text guard).
- File with caption → caption recorded; on injection the original file still routes through `_on_file_from_transport`.
- Bot's own messages → not recorded (is_own_bot guard).
- `last_llm_msg_id` evicted from the 200-msg ring → walk falls back to recording from buffer head. One-time over-inclusion; acceptable.
- Buffer empty (no chatter since last LLM call) → empty string, no block injected.
- Very long buffer text (>4000 chars) → oldest messages dropped with `[…older messages truncated…]` prefix.
- Sender's `handle` is None (e.g., Telegram users without a username) → falls back to `display_name`, then `"unknown"`.

**Tests:**
- 7 unit tests on `ChatHistory` (record + retrieve, mark + later retrieve excludes pre-mark, overflow eviction, char-cap truncation, DM skip, own-bot skip, empty-text skip).
- 3 bot-integration tests via `FakeTransport`: full group message → record → second group message triggers backend → backend gets the first as `recent_discussion`.
- 2 cross-backend tests: same recorded buffer; assert Claude's `--append-system-prompt` payload AND Codex's `<system-reminder>` block both contain the discussion text.
- 1 cross-transport type-level test: parametrized over `ChatRef(transport_id=t, …)` for `t in {fake, telegram, web}`; record + retrieve identical behavior for ROOM-kind refs. This does not imply the current Web UI emits ROOM-kind messages.
- 1 regression: `_on_file_from_transport` and `_on_voice_from_transport` also call `record`.

### Sub-feature 4 — `meta_dir` config field

**Goal:** Let operators put per-bot / per-plugin persistent storage on a different volume than `config.json`. Plugin-side machinery is already present in `plugin.py`; only the **`Config.meta_dir` config field** is missing.

**Change set:**

`config.py`:
- Module-level: `DEFAULT_META_DIR = DEFAULT_CONFIG.parent / "meta"` next to `DEFAULT_CONFIG`.
- New helper:
  ```python
  def resolve_project_meta_dir(meta_dir: Path, project_name: str) -> Path:
      """Return <meta_dir>/<project_name>/, creating parents idempotently."""
      path = meta_dir / project_name
      path.mkdir(parents=True, exist_ok=True)
      return path
  ```
- `Config.meta_dir: Path = field(default_factory=lambda: DEFAULT_META_DIR)` field.
- Loader: read `raw["meta_dir"]` as a string, `Path(...).expanduser()`. Missing → default. Non-string → log warning + default.
- Saver: emit `meta_dir` only when it differs from `DEFAULT_META_DIR` (emit-only-when-non-default; matches `respond_in_groups` policy).

`bot.py:_init_plugins`:
- Replace hardcoded `Path.home() / ".link-project-to-chat" / "meta" / self.bot_name` with `resolve_project_meta_dir(self._config.meta_dir, self.name)`.
- Pass into `PluginContext.data_dir`. Existing `Plugin.data_dir` property keeps working unchanged (it appends `/plugins/<plugin_name>/`).

**Backend / transport portability:** `Config.meta_dir` is a top-level field, completely independent of backend or transport. Used only by the plugin framework, which is already transport-portable.

**Tests:**
- Config loader: missing key → default. Custom string → loaded. Tilde expansion works.
- Config saver: default value → key absent. Custom value → key present.
- `resolve_project_meta_dir`: creates intermediate directories; idempotent on second call.
- `_init_plugins`: when `Config.meta_dir = /tmp/custom`, plugin `data_dir` resolves to `/tmp/custom/<project_name>/plugins/<plugin_name>/`.

## Cross-cutting concerns

**Backend portability matrix:**

| Sub-feature | Plumbing | Claude render | Codex render | Future backends |
|---|---|---|---|---|
| #1 Safety prompt | Instance field `BaseBackend.safety_system_prompt: str \| None` (per-bot, set at startup) | `--append-system-prompt` `parts` list | `<system-reminder>` block | Inherit field; implement render once |
| #3 Group context | Per-call kwarg `chat_stream(..., recent_discussion="")` (per-message, no shared state) | Same `parts` list, value from kwarg | Same `<system-reminder>` block, value from kwarg | Add kwarg to `chat_stream`; render in native style |

Both use the same per-backend render layer next to `team_system_note`. Documented inline in `backends/base.py`. Different plumbing because the lifecycle differs: safety is per-bot, recent_discussion is per-message.

**Transport portability matrix:**

| Sub-feature | Mechanism | Telegram | Web | Future Slack / Google Chat |
|---|---|---|---|---|
| #2 Hot-reload | Pure config-disk read; auth-only | Works | Works | Works |
| #3 Group context | `ChatRef.kind == ROOM` check; `ChatRef` is hashable; transport-portable | Works (groups + supergroups) | Deferred: current Web UI emits DM-kind direct chats; future Web rooms work once they emit ROOM refs | Will work (channels = rooms) |
| #4 meta_dir | Plugin framework; transport-agnostic | Works | Works | Works |

`tests/test_transport_lockout.py` will be extended to also lock down `chat_history.py` (zero `telegram.*` imports).

**Plugin framework interaction:**
- Plugins do not see `recent_discussion`. It's injected at the backend layer, after plugins' `on_message` hook has already fired. Plugins still see raw `IncomingMessage`s through their existing API.
- Plugins do see the new `safety_system_prompt` via the backend instance they're handed (read-only).
- Plugins do see `Config.meta_dir` via `PluginContext.data_dir`.

**Auth + role gates:**
- All four sub-features are additive; none change the role-enforcement surface.
- Hot-reload preserves the executor / viewer split: a viewer demoted on disk becomes a viewer in memory within 5 s.

## Acceptance criteria

A v1.2.0 deployment with all four sub-features enabled passes these checks:

1. **Safety prompt.** With `safety_prompt` unset (default) and Claude backend, the bot's stdout shows `--append-system-prompt` containing "Only make changes or run commands when explicitly asked". With `safety_prompt: ""`, it doesn't. With Codex, the analogous `<system-reminder>` block is present / absent. With both backends, asking the bot to "restart the service" produces a description-and-ask, not an action.
2. **Hot-reload.** Manager bot runs `/add_user`. Project bot's next auth attempt for the new user succeeds within 5 s, with the correct role, no restart.
3. **Group context.** In a Telegram group with `respond_in_groups=true`, three humans chat for two minutes between LLM invocations. When the bot is mentioned, its next reply demonstrates awareness of the human exchanges (e.g., "you were discussing X — for that, …"). The same `ChatHistory` code path is transport-portable for any future Web room surface that emits `ChatKind.ROOM`; the current Web UI is direct browser-to-bot chat and remains DM-kind.
4. **meta_dir.** Setting `meta_dir: /var/lib/lptc/data` and restarting moves plugin storage to `/var/lib/lptc/data/<project>/plugins/<plugin>/`. Existing plugins running before the change continue to use the old path until they restart.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Safety prompt makes the bot too cautious for power users | Per-project opt-out via `safety_prompt: ""`. Manager bot exposes a custom-string editor. |
| Hot-reload races with manager writes | `locked_config_rmw` on the manager side; `_auth_dirty` skip on the bot side; deterministic merge by username with locked_identities union. |
| Group context injects too much text | 200-msg ring buffer (matches GitLab); 4000-char serialized cap with oldest-first truncation. Worst case: ~10K tokens of context per invocation. |
| Group context leaks unauthorized-user text to LLM | Documented in CHANGELOG and README; matches GitLab; operator can revert by setting `respond_in_groups: false` (no LLM call in groups → no leak). |
| Plugin state stranded after `meta_dir` change | Operator-managed migration; documented in CHANGELOG. The old plugin's `data_dir` stops being accessed; new plugins use the new path. |

## Out of scope (deferred)

- Persistence of `ChatHistory` across restarts.
- Per-chat `recent_discussion` size override (use the global default).
- Per-plugin `data_dir` migration tooling.
- A production Web room model for multi-user room chatter. Current Web UI chats are `ChatKind.DM`; Web ROOM context starts when WebTransport has an actual room surface.
- Web-transport awareness in `_build_telegram_awareness` (renamed and generalized later — pre-existing tech debt, not v1.2.0's responsibility).
- A "describe-only" mode for the safety prompt (operator overrides text instead).
